# YiCai Review Learning Engine Design

Date: 2026-07-06

## Summary

Build a dual-track review learning system for simulated number generation.

The system will save both user-generated picks and system-generated benchmark picks, compare them with later official draw results, score them with lottery-specific rules, and surface strategy performance trends in the Generate Numbers page.

This is a data review and calibration feature. It must never claim that historical data can guarantee future results.

## Goals

- Persist every meaningful simulated pick so it can be reviewed after the target draw.
- Run a system benchmark batch for each lottery and strategy before the next draw.
- Review predictions automatically when new draw data arrives.
- Score Mark Six, Shuangseqiu, and Qixingcai with their own rules.
- Show users an understandable review panel: latest review, my history, system strategy ranking, and calibration notes.
- Prepare an adaptive strategy foundation without silently changing existing strategy behavior.

## Non-Goals

- No betting, purchase, or participation workflow.
- No guaranteed prediction claims.
- No opaque machine-learning model in the first version.
- No user account requirement in the first version.
- No retroactive scoring as if picks were made before past draws unless explicitly marked as backtest data.

## Existing Context

Backend already has:

- `Draw`: canonical draw table with `lottery_type`, `num1..num6`, `special_num`, and derived features.
- `FrequencyCache` and `PairFrequency`: strategy inputs for statistics and pair-chain generation.
- `GET /api/v1/analysis/generate`: simple strategy generator.
- `POST /api/v1/analysis/layered_pick`: layered funnel generator, including Qixingcai position-aware mode.
- Scrape and jackpot paths that keep draw data fresh.

Frontend already has:

- `GenerateNumbers.vue`: user-facing simulated pick page.
- `api.generate()` and `api.layeredPick()`.
- LocalStorage-only helper methods named user draws, but no backend review store.

## Design Decision

Use a dual-track review engine:

- `source_type = "user"` for picks generated by the user on the Generate Numbers page.
- `source_type = "system"` for scheduled benchmark picks generated by the backend.

Both tracks use the same storage and scoring tables, so the UI can compare them consistently.

## Data Model

### PredictionBatch

One generation action. A batch can contain one or more generated sets.

Fields:

- `id`
- `batch_key`: public stable identifier.
- `lottery_type`: `marksix`, `ssq`, `qxc`.
- `source_type`: `user` or `system`.
- `user_key`: anonymous local device key for user batches; null for system batches.
- `generator_mode`: `simple`, `layered`, `fortune`, or `system_benchmark`.
- `strategy`: `hot`, `weighted_random`, `overdue`, `balanced`, `pair_chain`, `layered`, `adaptive`.
- `strategy_version`: initial value `v1`.
- `params_json`: exact generation parameters and active lottery config.
- `target_draw_date`: the draw date this batch is meant to review against.
- `target_draw_number`: optional draw number when known.
- `context_json`: latest draw number, history window, frequency cache timestamp, and source endpoint.
- `created_at`
- `reviewed_at`
- `status`: `pending`, `reviewed`, `expired`, `invalid`.

### PredictionPick

One generated set inside a batch.

Fields:

- `id`
- `batch_id`
- `pick_index`
- `regular_numbers_json`: array. For Qixingcai this preserves position order and repeats.
- `special_number`: Mark Six special number, SSQ blue ball, or QXC back-zone digit.
- `display_json`: optional UI-ready formatting.
- `created_at`

### PredictionReview

One scored result for one prediction pick.

Fields:

- `id`
- `pick_id`
- `draw_id`
- `lottery_type`
- `score`
- `primary_hits`
- `special_hit`
- `exact_hit`
- `detail_json`
- `created_at`

### StrategyPerformanceSnapshot

Aggregated performance used for UI ranking and later adaptive weighting.

Fields:

- `id`
- `lottery_type`
- `source_type`: `user`, `system`, or `all`.
- `strategy`
- `window_size`: `10`, `24`, `50`, or `all`.
- `sample_count`
- `avg_score`
- `avg_primary_hits`
- `high_hit_count`
- `exact_hit_count`
- `special_hit_rate`
- `recommended_weight`
- `updated_at`

## Target Draw Resolution

When a user generates picks, the backend should assign a target draw:

- If the client sends `target_draw_date`, validate it against the lottery schedule and use it.
- Otherwise select the next known scheduled draw after `created_at`.
- If the draw number is not known yet, keep `target_draw_number` null and resolve it during review by matching `lottery_type + target_draw_date`.

When a system batch is created, it always targets the next scheduled draw for that lottery.

## System Benchmark Run

The backend should create benchmark batches with a deterministic envelope and random internal picks:

- For each lottery: `marksix`, `ssq`, `qxc`.
- For each strategy: `hot`, `weighted_random`, `overdue`, `balanced`, `pair_chain`, `layered`.
- For each strategy: generate 10 picks in V1.
- Save every batch with `source_type = "system"`.

The benchmark job should run in two places:

- After a successful new draw sync, review old pending predictions, then create benchmark batches for the next draw.
- Via an admin endpoint for manual backfill or recovery.

The job must be idempotent. It should not create duplicate system batches for the same `lottery_type + strategy + target_draw_date + strategy_version`.

## Review Trigger

After new draw data is inserted:

1. Find pending batches for the same lottery where `target_draw_date <= draw.draw_date`.
2. Match each batch to the draw with the same target date when possible.
3. Score each pick.
4. Mark the batch `reviewed`.
5. Recompute performance snapshots for affected lottery and strategies.
6. Create the next system benchmark batch if missing.

There should also be a manual endpoint:

- `POST /api/v1/review/run`
- Optional payload: `lottery_type`, `draw_id`, `target_draw_date`.

## Scoring Rules

### Mark Six

Draw shape:

- Main numbers: `num1..num6`
- Special number: `special_num`

Scoring:

- `primary_hits`: count intersection between predicted regular numbers and six main numbers.
- `special_hit`: predicted special equals draw special.
- `exact_hit`: `primary_hits == 6`.
- `score`: `primary_hits * 10 + special_hit * 3`.

Important rule:

- Mark Six first prize review is based on six main numbers. Special number is reviewed separately as an auxiliary signal.

### Shuangseqiu

Draw shape:

- Red balls: `num1..num6`
- Blue ball: `special_num`

Scoring:

- `primary_hits`: count intersection between predicted red balls and six red balls.
- `special_hit`: predicted blue equals draw blue.
- `exact_hit`: `primary_hits == 6 && special_hit == true`.
- `score`: `primary_hits * 10 + special_hit * 5 + exact_hit * 20`.

Important rule:

- Blue ball is not auxiliary. Jackpot-level exact hit requires six red balls plus the blue ball.

### Qixingcai

Draw shape:

- Position 1-6: `num1..num6`
- Back-zone position: `special_num`

Scoring:

- `primary_hits`: count exact position matches across positions 1-6.
- `special_hit`: predicted back-zone digit equals `special_num`.
- `exact_hit`: `primary_hits == 6 && special_hit == true`.
- `detail_json.digit_pool_hits`: optional count of digits appearing regardless of position, for analysis only.
- `score`: `primary_hits * 10 + special_hit * 8 + exact_hit * 30`.

Important rule:

- Qixingcai must preserve position order, allow repeats, and preserve `0`.
- Qixingcai picks must never be sorted or deduplicated.

## API Design

### Save User Batch

`POST /api/v1/review/predictions`

Purpose:

- Save generated picks after existing generation endpoints return.

Payload:

- `lottery_type`
- `source_type = "user"`
- `user_key`
- `generator_mode`
- `strategy`
- `params`
- `target_draw_date`
- `sets`: array of `{ regular: number[], special: number }`

Response:

- `batch_key`
- `target_draw_date`
- `target_draw_number`
- `saved_count`

### User Review History

`GET /api/v1/review/me?user_key=...&lottery_type=...`

Returns:

- Recent reviewed and pending user batches.
- Per-batch hit details.
- User aggregate stats for the current lottery.

### Strategy Performance

`GET /api/v1/review/strategies?lottery_type=...&source_type=system&window=24`

Returns:

- Strategy ranking.
- Sample count.
- Average score.
- Average primary hits.
- Special hit rate.
- High-hit count.
- Recommended weight.

### Latest Review

`GET /api/v1/review/latest?lottery_type=...`

Returns:

- Latest reviewed draw.
- Best system strategies for that draw.
- Distribution of user results if user samples exist.

### Manual Review Run

`POST /api/v1/review/run`

Returns:

- Reviewed batch count.
- Reviewed pick count.
- Snapshot count updated.
- System benchmark batches created.

## Frontend Design

Add a "Review Learning" section to `GenerateNumbers.vue`.

Placement:

- Below the generated result area.
- Above disclaimer and ad slot.

Sections:

- Latest Review: shows newest draw review and best-performing system strategy.
- My Picks: shows user's pending and reviewed picks for current lottery.
- System Benchmark: strategy leaderboard for last 24 reviewed draws.
- Calibration Notes: plain-language explanation of which strategy weight increased or decreased.

User key:

- Generate an anonymous local key in LocalStorage, separate from fortune user key.
- Do not require login.

Save behavior:

- After simple generation succeeds, call `POST /review/predictions`.
- After layered generation succeeds, save displayed combinations.
- If saving fails, keep showing generated numbers and display a small non-blocking warning.

## Adaptive Strategy Policy

V1 should not silently mutate existing strategy outputs.

Instead:

- Add an optional `adaptive` strategy after review data exists.
- `adaptive` chooses a base strategy using `recommended_weight`, then generates with that base strategy.
- Keep a minimum random exploration rate of 20% so the system does not overfit recent draws.
- Show the chosen base strategy in the result, for transparency.

## Compliance Copy

Every review panel must include compact copy:

> Review data only compares simulated picks with historical draw results. Lottery draws are random; historical performance cannot guarantee future results. This is for entertainment and data review only.

Chinese UI copy should say:

> 复盘数据仅用于对照历史开奖结果。彩票开奖具有随机性，历史表现不能保证未来结果。本功能仅供娱乐和数据复盘参考。

## Error Handling

- If no target draw can be resolved, save the batch as `pending` with a null target number and a validated target date.
- If the draw is later missing, keep the batch pending.
- If a saved pick has invalid number shape, reject the request with a clear validation error.
- If review scoring fails for one pick, mark that pick invalid and continue the batch.
- If performance snapshot recompute fails, do not roll back already-scored reviews; log the failure and allow manual recompute.

## Privacy

- Store only an anonymous device key for user batches.
- Do not store IP address.
- Do not require name, phone, email, or login.
- If user clears local storage, future batches start under a new anonymous key.

## Testing Plan

Backend:

- Unit-test Mark Six scoring: 6 main hits with missed special still exact for main-number review.
- Unit-test SSQ scoring: 6 red without blue is not exact hit.
- Unit-test QXC scoring: position order, duplicate digits, and `0` are preserved.
- Unit-test idempotent system benchmark creation.
- Unit-test pending batch review after inserting a new draw.

Frontend:

- Generate simple strategy and confirm save call happens.
- Generate layered strategy and confirm displayed combinations are saved.
- Confirm save failure does not block number display.
- Confirm review panel renders empty, pending, and reviewed states.
- Check desktop and mobile layout for no nav overlap.

Integration:

- Trigger a system benchmark for one lottery.
- Insert or use a known draw.
- Run review.
- Confirm strategy snapshot updates.

## Rollout Plan

Phase 1:

- Add models, scoring service, review router, and manual review endpoint.
- Save user-generated picks from the frontend.
- Add basic review panel.

Phase 2:

- Hook review execution into successful draw sync.
- Add system benchmark creation for next draw.
- Add strategy performance leaderboard.

Phase 3:

- Add optional `adaptive` strategy using performance snapshots.
- Add admin/backfill controls.
